# Scan Command Overhaul — Low-Level Design

## Architecture

### Current shape (baseline)

- Command dispatch: `main()` on `os.Args[1]` (`main.go:45-111`); `scan`/`rebuild`/
  `index` → `cmdScan(target)` where target is `"all"` or the first arg
  (`main.go:61-66`).
- `cmdScan` (`main.go:525-563`): `os.Remove(dbFile)` → recreate schema →
  `FullScan` each (optionally filtered) repo → store `git_commit_<name>` per repo
  → store one global `meta["last_scan"]`.
- Repos are stored in a flat file `~/.local-search/repos`, one pipe-delimited line
  `name|path|skipdirs` (`main.go:2199-2265`). `repoEntry{Name,Path,SkipDirectories}`.
- `repo add` / `repo remove` call `cmdScan("all")` (`main.go:172`, `:280`).
- `repo list` prints name + path only (`main.go:283-292`).
- CWD→repo resolver `scope.NearestRepoForCWD(cwd, repos)` exists
  (`scope/scope.go:357-387`) — longest enclosing registered path, `ok=false` if
  none — used by search but not by scan.
- Per-repo surgical delete primitive `localdb.DeleteRepo(db, repoName)` exists
  (`db/index.go:364`): deletes that repo's rows from `specs`, `spec_vectors`,
  `spec_tags`, `spec_edges`, and rebuilds `specs_fts`.
- `meta` get/set: `localdb.GetMeta` / `SetMeta` (`db/schema.go:228-239`).
- Incremental update at query time: `ensureDB()` (`main.go:575-633`) and
  `runIncrementalUpdates()` (`main.go:1624-1671`) call `localdb.IncrementalScan`
  and rewrite `git_commit_<repo>`.

### Target shape

**1. Scan target resolution (new function `resolveScanTarget`)**

Introduce a resolver that maps the invocation to a concrete list of repos plus a
mode:

```
resolveScanTarget(args, cwd, repos) -> (mode, targets, err)
  args == ["all"]                    -> (ModeFullRebuild, all repos, nil)
  args == ["<name>"] and name known  -> (ModeSurgical,   [named repo], nil)
  args == ["<name>"] and name unknown-> error "unknown repo <name>"
  args == []                         -> NearestRepoForCWD(cwd, repos):
                                          ok   -> (ModeSurgical, [enclosing repo], nil)
                                          !ok  -> error "not inside a registered repo; cd into one or run 'scan all'"
```

`main.go:61-66` dispatch is updated to pass the raw args (not a pre-baked
`"all"`) into `cmdScan`, which now calls `resolveScanTarget` first and returns
**before mutating anything** on error. Note `cmdScan`'s signature change ripples
to its **internal callers**: `ensureDB` (`main.go:577`) and the `repo add`/`repo
remove` paths call it too — all call sites must migrate. `ensureDB`'s
"DB missing → rebuild all" bootstrap keeps invoking the full-rebuild mode
explicitly (equivalent to `scan all`).

**2. Surgical vs. full scan (`cmdScan` rewrite)**

- `ModeFullRebuild` (`scan all`, and schema-version reset path): keep today's
  behavior — `os.Remove(dbFile)` → recreate schema → `FullScan` every repo. This
  is the **only** path that deletes the DB file.
- `ModeSurgical` (default / `scan <name>` / automation): **do not** remove the DB.
  Open (or bootstrap) the DB, then for the single target repo:
  1. `localdb.DeleteRepo(db, repo.Name)` — clear just this repo's rows.
  2. `localdb.FullScan(db, repo.Name, repo.Path, repo.SkipDirectories)` —
     re-index just this repo.
  3. If git: `SetMeta("git_commit_"+name, git.CurrentCommit(path))`.
  4. Record per-repo timestamps (below).
  Steps 1–2 must be **atomic to concurrent readers** (R-2.8): a `search` running
  during the scan must see either the old or the new index for this repo, never
  the empty window between delete and re-insert. Since automation fires this path
  on every git event — exactly when the user is likely querying — the naive
  two-transaction form is not acceptable. Wrap delete+reindex in a single
  transaction, or build into a staging area and swap; the exact mechanism is an
  implementation choice constrained by `FullScan`'s current transaction
  boundaries, but the observable guarantee is fixed.
  If the DB file does not exist yet, a surgical scan first creates the schema
  (bootstrap), then indexes only the target repo — it never fans out to all
  repos.

**3. Per-repo state tracking**

Storage split by durability requirement:

| Field | Store | Key / field | Set by | Survives `scan all`? |
|---|---|---|---|---|
| Date added | flat repos file | new 4th field `added_at` (RFC3339) | `repo add` | ✅ (file, not DB) |
| Last scan | `meta` table | `last_scan_<name>` | surgical + full scan | ♻ regenerated by scan |
| Last index update | `meta` table | `last_index_update_<name>` | `IncrementalScan` path | ♻ (empty until next update) |
| Latest indexed commit | `meta` table | `git_commit_<name>` (exists) | scan + incremental | ♻ regenerated by scan |

- Flat-file format extends to `name|path|skipdirs|added_at`. `parseRepoEntryLine`
  changes `SplitN(...,3)` → `SplitN(...,4)`; a 4th field, when present and
  parseable as RFC3339, populates `repoEntry.AddedAt`. Legacy 2/3-field lines
  parse as today with `AddedAt` empty. `repoEntry` gains `AddedAt string`.
  **`added_at` is positional (the 4th field), so `formatRepoEntryLine` MUST NOT
  omit an empty skip-dirs field when `added_at` is present** — it emits an empty
  3rd field placeholder: `name|path||<added_at>` (R-6.6). Omitting it (today's
  "append skipdirs only when non-empty" logic at `main.go:2216-2224`) would push
  the timestamp into `parts[2]`, where `normalizeSkipDirectoryNames` rejects it
  and the entire line is dropped — silently losing the repo. This is the single
  most important encoding rule in this change.
- **Downgrade hazard (documented, not mitigated):** an older binary reading a new
  4-field line does `SplitN(...,3)`, folding `added_at` into the skip-dirs field,
  which is then rejected and the line dropped. Acceptable for a single-binary
  user; note it near the repos-file format docs.
- The global `meta["last_scan"]` is retained for `stats` backward-compat but is
  no longer the source of per-repo scan time.
- The per-repo "last index update" write is added in the incremental path right
  after a successful `IncrementalScan` that changed files. The incremental logic
  is duplicated in **both** `ensureDB` (`main.go:614-630`) and
  `runIncrementalUpdates` (`main.go:1653-1668`); to avoid the timestamp drifting
  between the two sites, extract a shared helper that performs the
  `IncrementalScan` + `git_commit_<name>` + `last_index_update_<name>` writes and
  call it from both (R-3.4).

**4. `repo list` display**

`repoList` (`main.go:283-292`) opens the DB read-only (best-effort; tolerate a
missing DB) and reads the three meta keys per repo, plus `added_at` from the flat
entry. New columnar output:

```
NAME                 ADDED        LAST SCAN     LAST UPDATE   COMMIT   PATH
docs                 2026-07-18   2h ago        10m ago       3231af4  /Users/.../docs
```

Timestamps rendered via the existing `humanAge` helper (`main.go:507-524`);
missing values render as `—`. Commit shown short (first 7). If the DB is absent,
timestamp/commit columns show `—` and only name/path/added are populated.

**5. Configurable automation — `scan-hooks` command**

New subcommand pair modeled on `cmdInstallSkill` (`skill.go`):

```
local-search scan-hooks install   [--mechanism git-hooks,shell] [--force]
local-search scan-hooks uninstall [--mechanism git-hooks,shell]
```

- Operates on the **CWD-resolved repo** (`NearestRepoForCWD`); errors if not
  inside a registered repo (same guard as `scan`).
- `--mechanism` selects what to install (comma list). If omitted, install prompts
  interactively (reuse the existing `ui.go` selection helpers) listing the
  available mechanisms; the user picks one or both.
- **git-hooks mechanism**: write managed hook scripts into `<repo>/.git/hooks/`
  for `post-merge`, `post-checkout`, and `post-rewrite` — the history-movement
  events (pull/merge/rebase/checkout) the query-time incremental path is weakest
  at surfacing promptly. **`post-commit` is excluded** (redundant with the
  query-time incremental path; highest scan-storm risk on rapid commits) (R-5.4).
  Each hook is a small `#!/bin/sh` that:
  - runs the change-gate first (skip if no spec files changed vs. last indexed
    commit) and only then dispatches `local-search scan <name>` surgical (R-5.11),
  - **exits 0 unconditionally and dispatches the scan detached/backgrounded**, so
    a slow or failing scan never blocks or fails the git operation (R-5.10),
  - relies on a per-repo lock so overlapping triggers don't run concurrent scans
    (R-5.12).
  Hooks use a sentinel-delimited managed block so installing does not clobber a
  pre-existing user hook (merge inside the block, or create the file if absent),
  and uninstalling removes only the managed block, deleting the file only if it
  becomes empty. Existing hooks are made executable (`0755`); `--force` overwrites
  a stale managed block. If the resolved repo is **not** a git repository, the
  git-hooks mechanism is skipped with a message and other requested mechanisms
  still install (R-5.4a).
- **shell mechanism**: emit a shell snippet (a `chpwd`/prompt hook function) that
  runs a surgical scan when the user `cd`s into a registered repo, behind the same
  change-gate (R-5.11) and per-repo lock (R-5.12) so navigation does not trigger
  repeated full re-walks. Install writes the snippet to a known file (e.g.
  `~/.local-search/shell-hook.sh`) and prints the one-line `source` directive for
  the user's shell rc (it does not silently edit rc files); uninstall removes the
  snippet file and prints the line to delete. The snippet is generated inline (no
  new embedded asset required).
- The hook/shell scripts invoke the same binary already on `PATH`; the resolved
  repo name is baked into the generated script so the trigger is unambiguous.

**6. Internal caller updates**

- `repo add` (`main.go:135-173`): after appending the entry (now with
  `added_at = now`), run a **surgical** scan of just the new repo instead of
  `cmdScan("all")`.
- `repo remove` (`main.go:~280`): call `localdb.DeleteRepo(db, name)` (surgical)
  instead of `cmdScan("all")`; drop the flat entry as today.

### Data flow (default `scan`)

```
user runs `scan` in /Users/me/docs/sub
  -> cmdScan([])            (main.go dispatch, raw args)
  -> resolveScanTarget([], cwd, repos)
       -> NearestRepoForCWD -> repo "docs" @ /Users/me/docs
  -> ModeSurgical, targets=[docs]
  -> openDB (no os.Remove)
  -> DeleteRepo(db,"docs") -> FullScan(db,"docs",path,skip)
  -> SetMeta("git_commit_docs", HEAD)
  -> SetMeta("last_scan_docs", now)
  (repos B,C untouched)
```

## Constraints

- **Language/stack**: Go, single static binary; SQLite via the existing `db`
  package; no new third-party dependencies (git hooks are shell scripts; shell
  hook is a generated snippet). fsnotify is explicitly excluded.
- **Durability**: `added_at` must survive a full `scan all` (which deletes the DB
  file), so it cannot live in any DB table — it lives in the flat repos file.
- **Backward compatibility**: existing `~/.local-search/repos` files use 2/3-field
  lines and must continue to parse; existing databases must not require a manual
  migration (surgical scan bootstraps schema if the DB is absent).
- **Non-destructive default**: the default `scan` must make no mutation when it
  errors (outside-any-repo case) — resolution happens before any `os.Remove` or
  DB write. This inverts today's "delete DB first, then filter" ordering.
- **Hook safety**: installing/uninstalling git hooks must never destroy
  user-authored hook content; managed content is delimited by sentinels and
  removed surgically.
- **Cross-platform**: git hooks are POSIX `sh`; the shell hook targets the user's
  interactive shell. Windows shell-hook support is best-effort / documented,
  consistent with the existing `ui_windows.go` split.
- **Concurrency**: automation makes a second process writing the DB routine while
  a foreground `search`/`ensureDB` may also be writing. SQLite permits one writer;
  the DB must be opened with a bounded `busy_timeout` and/or WAL mode so a scan
  contending with a read does not error or corrupt the index (R-2.7). Combined
  with the per-repo automation lock (R-5.12), this bounds contention to
  DB-level serialization rather than hard failures.

## Key Decisions

1. **Retain `scan all` as the explicit full-rebuild keyword.** The user selected
   "CWD single-repo default". A full teardown+rebuild must still be reachable
   (schema resets, "rebuild everything" recovery). `all` is that keyword and is
   the only DB-deleting path. *Rejected*: removing full-rebuild entirely (leaves
   no recovery/reset path); making it implicit (surprising and destructive).
2. **Surgical scan reuses `DeleteRepo` + `FullScan`.** These primitives already
   exist and are per-repo correct; no new indexing code. *Rejected*: writing a
   bespoke per-repo diff-and-patch (redundant with `IncrementalScan`, higher risk).
3. **`added_at` in the flat file, other timestamps in `meta`.** Only the flat
   file survives a full rebuild; runtime timestamps are cheap to regenerate and
   acceptably reset on `scan all`. *Rejected*: a new non-derived DB table for
   added_at (still lost when `scan all` deletes the DB file); a sidecar JSON
   (redundant second config store).
4. **`scan-hooks install/uninstall` as a dedicated command pair** mirroring
   `install-skill`, operating on the CWD repo. *Rejected*: folding into
   `repo add` flags (couples registration with automation, no per-mechanism
   uninstall); a single interactive `install-automation` (chosen UX is the
   command pair, with interactive selection only when `--mechanism` is omitted).
5. **Automation triggers the surgical single-repo scan** (user's choice), not the
   incremental-only path — so non-git spec edits are also captured — and not a
   full rebuild — which would be heavy on every git event.
6. **Only git-hooks and shell mechanisms** (user's choice); no watcher daemon,
   avoiding process-lifecycle/PID complexity.
7. **`repo add`/`repo remove` become surgical.** Directly implied by the surgical
   decision; otherwise `repo add` would still wipe the whole DB, contradicting the
   goal. Kept in scope because it shares the exact primitive.

## Out of Scope

- File-system watcher / background daemon.
- Remote/push detection, `git fetch`, branch-awareness in change detection.
- Changes to spec-file globs, extraction, embeddings, or search ranking.
- Automatic editing of the user's shell rc files (we print the `source` line).
- Retroactively back-filling `added_at` for repos registered before this change
  (they render `—`).
- Any change to the global `meta["last_scan"]` semantics consumed by `stats`
  (retained as-is for compatibility).
